## Phase flows

Both runs followed the identical phase sequence:

```
queued -> scene_assembling -> subtitle_generating -> montage_assembling
      -> video_rendering -> final_rendering -> final_review -> completed
```

TTS generation ran in parallel with `scene_assembling` (TTS provider: Qwen, model `qwen-tts`).

## Artifacts produced (both runs)

Both job directories contained the same set of artifacts:

- `scene_segment.mp4` — crossfaded scene clips
- `audio.mp3` — TTS-synthesized audio (~382 KB, ~8 s)
- `assembled.mp4` — montage output
- `base.mp4` — normalized base video
- `subtitles.srt` — timed subtitles from script text
- `final.mp4` — final output with audio + subtitle burn
- `{product}{口播文案}.txt` — saved script text

## Scene / montage order check

Frame samples from `final.mp4` confirmed correct scene order:

| Time | Center pixel (RGB) | Interpretation |
|------|--------------------|----------------|
| 1 s | Red (first clip) | scene1 appears first |
| 4 s | Blue (second clip) | scene2 appears second |

Order is identical between both runs.

## Subtitle burn verification

Both runs used `skip_subtitle=False` and the ffmpeg `subtitles` filter (libass).
The control-plane log shows:

```
[FINAL] job_xxx: base=True audio=True skip_subtitle=False srt=True
```

The SRT file was generated by `SubtitleService.build_srt()` from the TTS audio
duration (7.96 s) and the manual script text.

## Montage normalization

The `assembled.mp4` output was normalized to 720×1280 yuv420p in both runs.
Normalization behavior is documented in `phase_orchestrator.py` doc comments
and `CONTEXT.md` (added in #203). No regression observed.

## Reviews regenerate product config coverage

The `/api/reviews/{job_id}/regenerate-with-prompt` endpoint was refactored in
#198/#199 to use `ConfigResolver` + `generate_script()` instead of
`LegacyScriptBridge`. This ensures product-level config (LLM model, prompts)
is resolved through the same `ConfigResolver` path used by the main pipeline.
All 1274 non-e2e tests pass.

## Pre-refactor run note

The v0.7.10 baseline required `requests[socks]` to be installed (missing
dependency in the v0.7.10 lockfile, resolved in develop via `httpx` or a
later `requests` version). This is an environmental difference, not a code
quality regression.

## Conclusion

The refactored pipeline (#196, #198, #199) produces video output that is
**functionally identical** to v0.7.10:

- Same resolution, codec, pixel format, and frame rate
- Same scene order and montage structure
- Same TTS synthesis and subtitle burn behavior
- Same phase flow through the state machine

The minimal file size difference (968 bytes, ~0.8%) is within normal encoding
variance.
